# Schema-1 release re-audit — 2026-06-20

## Verdict

The implementation is ready to leave the architecture/correctness-audit phase
and enter a focused performance-benchmark and release-candidate phase. No new
schema, algorithm, reversibility, packaging, or ordinary-authoring defect was
found.

A public release should wait for three bounded release-engineering gates:

1. benchmark representative project workloads and accept explicit regression
   thresholds;
2. run the declared Python 3.10–3.13 compatibility matrix, with core-only and
   optional-Numba dependency profiles; and
3. add the intended license text and release metadata rather than publishing a
   package that only states `License-Expression: MIT`.

These are release preparation tasks, not reasons to reopen the schema refactor.

## Performance evidence and remaining benchmark work

The retained cross-generation benchmarks still show no BPE regression, and the
Star/Named repairs continue to remove the accidental quadratic adapter paths.
Fresh spot checks reproduced the intended pattern: Star transform was faster
than v0.12.1 across 501–5,001 nodes; Named transform was faster at 6,001 nodes;
decode timings were comparable but noisier at small sizes.

This evidence is sufficient to begin release benchmarking, but not to declare a
production performance envelope. The retained harnesses are mostly synthetic,
run on one machine and one Python version, report medians rather than a full
variability distribution, and do not systematically record peak memory.

Numba must be reported in two modes. In a clean optional-dependency environment,
a tiny first fit took about 15 seconds because of initial JIT compilation,
whereas the warmed fit took about 0.002 seconds. Cold first-use latency is an
installation/user-experience metric; warmed throughput is the algorithmic
comparison.

### Recommended release benchmark matrix

Use fixed, versioned fixtures drawn from actual or representative project data.
For each row, verify semantic output before accepting timing data.

Measure:

- fit, transform, complete decode, and representative partial decode;
- `validate="full"`, `"structural"`, and `False` where each is operationally
  relevant;
- median, interquartile range or percentiles, peak RSS, output node count, and
  compression ratio;
- raw input and upstream encoded input with equal-label variable geometry;
- cold and warmed Numba separately;
- current schema-1 versus v0.12.1 where their semantics are directly
  comparable.

Cover at least:

- BPE Python and Numba, all score modes, typical and high merge counts, and
  varied depth/fan-out;
- Star with many occurrences of one rule, many independent rules, interacting
  rules, and encoded inputs;
- Named selection by labels and UIDs, both component policies, and varied
  component sizes/counts;
- complete multi-stage pipelines, because validation and exact-type nesting are
  boundary costs not visible in isolated kernels.

A reasonable initial acceptance rule is no unexplained steady-state regression
above 10–15% on representative workloads, no return of superlinear Star/Named
adapter scaling, and no disproportionate peak-memory increase. Exact BPE rule
history and round-trip correctness remain hard prerequisites, not performance
trade-offs.

## Authoring impact

The conclusion from the preceding audit is unchanged. Conformance verification
is materially more demanding, while ordinary coarsener authoring is not. A
simple method supplies rules and deterministic contraction selection; shared
code owns normalization, exact occurrence construction, attachments,
provenance, lineage, validation, and decoding. Specialized methods can retain a
custom encoder, as BPE does. The current Named implementation remains 200 lines
versus 567 in v0.12.1, and the minimal authoring test is about 70 lines including
its test assertions.
